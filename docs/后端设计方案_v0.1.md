## 📋 文档概述

本文档详细阐述 Rig-Core 的架构设计、核心模块、设计模式和最佳实践。Rig 是一个专注于人体工程学和模块化的 Rust 库，用于构建基于大语言模型（LLM）的应用程序。

**本方案的核心目的**：为外部系统提供统一、标准化的 LLM 能力接口，基于 HTTP 协议提供 REST API 服务和 MCP 协议服务，支持流式（SSE）和非流式两种传输模式，实现 LLM 服务和 MCP 工具的高可用、高性能对外输出。

### 文档目标

- 深入理解 Rig-Core 的架构设计理念
- 掌握各核心模块的功能和交互方式
- 学习如何扩展和集成新的模型提供商
- 了解最佳实践和设计模式
- 理解如何基于 Rig-Core 构建对外 API 服务

## 🎯 项目定位与核心价值

Rig-Core 是一个**生产级别的 LLM 应用开发框架**，旨在：

1. **统一多个 LLM 提供商**：通过统一接口支持 20+ 个主流 LLM 提供商

4. **类型安全**：充分利用 Rust 类型系统保证编译时安全
5. **异步优先**：全面支持异步操作，高性能并发

### 核心价值主张

**关键特性**：
- ✅ 完整的 LLM 补全和嵌入工作流支持
- ✅ 20+ 模型提供商统一接口
- ✅ 10+ 向量存储集成
- ✅ Agent 工作流和多轮对话
- ✅ 流式响应处理
- ✅ 工具调用（Function Calling）
- ✅ RAG（检索增强生成）支持
- ✅ 转录、音频生成、图像生成能力
- ✅ WASM 兼容
- ✅ 完整的 OpenTelemetry 支持

### 层次职责说明

#### 1. API 层（HTTP REST API 服务层 + MCP 协议服务层）

**核心职责**：向外部系统提供基于 HTTP 协议的标准化 API 接口

**接口分类**：

| 接口类型 | 端点前缀 | 协议标准 | 主要用途 |
|---------|---------|---------|---------|
| LLM REST API | `/v1/*` | OpenAI API 标准 | LLM 对话、补全、嵌入等 |
| MCP 协议接口 | `/mcp` | MCP 1.0 (JSON-RPC 2.0) | 工具调用、资源访问 |

**传输模式**：

| 模式 | 协议 | 使用场景 | 响应方式 |
|------|------|---------|---------|
| 非流式传输 | HTTP/1.1 | 短文本生成、嵌入、图像生成 | 一次性返回完整 JSON 响应 |
| 流式传输（SSE） | Server-Sent Events | 长文本生成、实时对话、MCP 工具调用 | 逐块推送响应内容（事件流） |

**LLM REST API 特点**：
- **统一端点**：同一个 API 端点支持流式和非流式模式（通过 `stream` 参数控制）
- **兼容标准**：遵循 OpenAI API 规范，便于客户端集成
- **协议简单**：仅使用 HTTP 协议，降低实现和维护复杂度
- **性能优异**：支持 HTTP/2，连接复用，减少握手开销

**MCP 协议接口特点**：
- **标准协议**：完全遵循 MCP 1.0 标准规范（JSON-RPC 2.0）
- **双向能力**：既是 MCP Client（调用外部 MCP 服务），也是 MCP Server（对外提供工具）
- **工具生态**：内置工具 + 代理转发外部 MCP 服务 + 自定义插件
- **互操作性**：任何 MCP 客户端（Python、TypeScript、Rust SDK）都可接入
- **流式支持**：通过 SSE (Server-Sent Events) 实现流式工具调用和进度通知
- **会话管理**：支持有状态的会话管理，保持客户端连接状态

**关键特性**：
- ✅ 统一接口：屏蔽底层不同 LLM 提供商的差异
- ✅ 双协议支持：REST API（LLM 调用）+ MCP 协议（工具调用）
- ✅ 流式响应：基于 SSE 的统一流式传输方案
- ✅ 工具生态：融合内部工具和外部 MCP 服务
- ✅ 会话管理：支持有状态的 MCP 会话和无状态的 REST 调用
- ✅ 负载均衡：支持水平扩展和请求分发
- ✅ 容错降级：故障转移、超时重试、熔断保护
- ✅ 可观测性：全链路追踪、指标监控、日志聚合

#### 2. 鉴权层（Authentication Layer）

**核心职责**：在 API 层和业务逻辑层之间提供统一的认证、授权和流量控制

**鉴权方式**：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| API Key | Bearer Token 认证 | 服务间调用、开发者接入 |
| JWT Token | JSON Web Token | 用户身份认证、会话管理 |
| OAuth 2.0 | 第三方授权 | 开放平台、第三方应用 |

**核心功能**：

1. **身份认证**：验证请求方身份的合法性
2. **权限控制**：基于角色/资源的访问控制（RBAC）
3. **速率限制**：防止滥用，保护系统稳定性
4. **租户隔离**：多租户环境下的数据和资源隔离
5. **审计日志**：记录所有访问行为，支持安全审计

**限流策略**：

| 维度 | 说明 | 示例 |
|------|------|------|
| 用户级别 | 按 API Key 限流 | 每分钟 60 次请求 |
| IP 级别 | 按来源 IP 限流 | 每分钟 100 次请求 |
| 端点级别 | 按 API 端点限流 | /v1/chat/completions 每秒 10 次 |
| 模型级别 | 按使用模型限流 | GPT-4 每分钟 20 次请求 |

**设计优势**：
- ✅ 层次分离：认证逻辑与业务逻辑解耦
- ✅ 统一管理：所有 API 端点共享同一套鉴权机制
- ✅ 灵活扩展：支持多种认证方式，易于添加新的认证策略
- ✅ 性能优化：token 缓存、session 复用，降低认证开销

#### 3. 数据层（Data Layer）

**核心职责**：提供统一的数据持久化、缓存和管理能力

**数据存储模块**：

| 模块 | 功能 | 存储方案 | 说明 |
|------|------|---------|------|
| 对话历史存储 | 保存多轮对话记录 | 关系型数据库 / NoSQL | 支持上下文管理和历史回溯 |
| 用户数据管理 | 用户信息、偏好设置 | 关系型数据库 | 用户画像、权限配置 |
| 向量数据存储 | 嵌入向量、知识库 | 向量数据库 | 支持 RAG 场景 |
| 配置管理 | 系统配置、模型参数 | 配置中心 / 数据库 | 动态配置、热更新 |
| 日志归档 | 请求日志、审计日志 | 时序数据库 / 对象存储 | 长期存储、合规审计 |
| 缓存层 | 热数据缓存 | Redis / Memcached | 提升响应速度 |

**核心功能**：

**1. 对话历史管理**
- **会话管理**：创建、查询、删除会话
- **消息存储**：保存用户消息和助手响应
- **上下文检索**：快速获取历史对话上下文
- **数据过期**：自动清理过期对话（可配置保留期限）

**2. 用户数据管理**
- **用户信息**：基本信息、认证凭证
- **偏好设置**：模型偏好、参数默认值
- **使用统计**：Token 使用量、成本统计
- **配额管理**：每日/每月使用限额

**3. 向量数据存储**
- **文档向量化**：存储文档嵌入向量
- **相似度检索**：向量相似度搜索
- **知识库管理**：文档增删改查
- **索引优化**：向量索引构建和维护

**4. 配置管理**
- **系统配置**：API 端点、超时设置
- **模型配置**：可用模型列表、参数范围
- **提示词模板**：预定义提示词库
- **动态更新**：无需重启的配置热更新

**5. 缓存策略**
- **响应缓存**：相同请求的响应缓存
- **嵌入缓存**：文本嵌入向量缓存
- **用户信息缓存**：减少数据库查询
- **过期策略**：LRU/TTL 过期机制

| 数据类型 | 推荐方案 | 备选方案 |

## 🌐 API 层设计方案

### HTTP REST API 服务

#### 接口设计规范

**遵循标准**：
- 遵循 OpenAPI 3.0 规范
- 兼容 OpenAI API 标准
- 支持流式响应（Server-Sent Events）

**传输模式设计**：

系统支持两种传输模式，通过同一端点的 `stream` 参数进行控制：

| 传输模式 | stream 参数 | Content-Type | 响应特点 | 使用场景 |
|---------|------------|--------------|---------|---------|
| 非流式传输 | false | application/json | 完整响应一次返回 | 短文本、嵌入、图像 |
| 流式传输 | true | text/event-stream | 逐块推送（SSE） | 长文本、实时对话 |

**核心端点**：

**LLM 核心接口**：

| 端点 | 方法 | 功能 | 流式支持 | 说明 |
|------|------|------|---------|------|
| `/v1/chat` | POST | 聊天补全 | ✅ SSE | 核心对话接口 |
| `/v1/generations` | POST | 文本补全 | ✅ SSE | 文本续写接口 |
| `/v1/embeddings` | POST | 文本嵌入 | ❌ | 向量生成接口 |
| `/v1/models` | GET | 获取模型列表 | ❌ | 模型查询接口 |
| `/v1/images/generations` | POST | 图像生成 | ❌ | 图像生成接口 |
| `/v1/audio/transcriptions` | POST | 音频转录 | ❌ | 语音转文字 |
| `/v1/audio/speech` | POST | 文字转语音 | ✅ SSE | 文字转语音 |

**MCP 协议接口**（遵循标准 MCP 协议）：

| 端点 | 方法 | 功能 | 流式支持 | 说明 |
|------|------|------|---------|------|
| `/mcp` | POST | MCP JSON-RPC 端点 | ✅ SSE | 标准 MCP 协议入口（支持普通请求和 SSE 流式） |

#### 非流式传输模式

**请求特点**：
- 标准 HTTP POST 请求
- `stream: false` 或不设置 stream 参数
- Content-Type: `application/json`

**响应特点**：
- 等待完整结果生成后一次性返回
- 响应格式为完整的 JSON 对象
- 适合短文本、嵌入向量等快速返回的场景

**优势**：
- 简单易用，标准 HTTP 客户端即可调用
- 完整的响应结构，便于解析
- 适合批量处理和离线任务

**劣势**：
- 首字节延迟高（需等待全部生成完成）
- 长文本生成时用户体验较差
- 占用连接时间长

#### 流式传输模式（SSE）

**请求特点**：
- 标准 HTTP POST 请求
- `stream: true` 参数
- Content-Type: `application/json`

**响应特点**：
- Content-Type: `text/event-stream`
- 使用 Server-Sent Events (SSE) 协议
- 响应数据逐块推送
- 每个数据块为独立的 JSON 对象
- 以 `data: [DONE]` 标记结束

**SSE 格式**：
```
data: {"choices":[{"delta":{"content":"你"}}]}

data: {"choices":[{"delta":{"content":"好"}}]}

data: [DONE]
```

**优势**：
- ✅ 低延迟：立即返回首个 token
- ✅ 实时体验：用户可见文本逐步生成
- ✅ 用户友好：类似打字机效果
- ✅ 可中断：客户端可随时断开连接
- ✅ 标准协议：SSE 是 HTML5 标准，浏览器原生支持

**劣势**：
- 需要客户端支持 SSE 协议
- 连接保持时间较长
- 服务器需要维护更多连接状态

#### MCP 协议对外接口设计

**设计理念**：提供标准的 MCP (Model Context Protocol) 服务端接口，让外部客户端可以通过标准 MCP 协议访问 Rig 的工具生态。

**协议规范**：
- **基础协议**：JSON-RPC 2.0
- **传输方式**：HTTP POST（普通请求）+ SSE（流式响应）
- **协议版本**：MCP 1.0
- **内容类型**：application/json（请求/非流式响应）、text/event-stream（SSE 流式响应）

**架构设计**：

```
外部 MCP 客户端（Python/TypeScript/Rust SDK）
         ↓
    HTTP POST /mcp
         ↓
    ┌──────────────────┐
    │  请求处理路由    │
    └────┬────────┬────┘
         │        │
    非流式│        │流式（建立 SSE 连接）
         │        │
         ↓        ↓
    [JSON]    [SSE Stream]
         │        │
         └────┬───┘
              ↓
    鉴权层（API Key / JWT Token）
              ↓
    MCP Server 实现（Rig）
              ↓
    工具路由器（Tool Router）
              ↓
┌─────────────┬──────────────┬───────────────┐
│ 内置工具    │ MCP Client   │  自定义工具   │
│ (Rust 实现) │ (代理转发)   │  (插件系统)   │
└─────────────┴──────────────┴───────────────┘
              ↓
    会话管理器（Session Manager）
              ↓
    返回 JSON-RPC 响应
```

**传输模式说明**：

| 模式 | HTTP 方法 | Content-Type | 响应方式 | 使用场景 |
|------|----------|--------------|---------|---------|
| 非流式 | POST | application/json | 一次性返回完整 JSON | 短任务、工具列表查询 |
| 流式（SSE） | POST | text/event-stream | 逐块推送事件 | 长时间工具调用、进度通知 |

**MCP 端点**：

**端点**：`POST /mcp`

**功能**：标准 MCP 协议入口，支持所有 MCP 方法和流式/非流式响应

**请求处理流程**：

```
客户端发送 JSON-RPC 请求
    ↓
服务端解析请求
    ↓
判断方法是否支持流式
    ↓
    ├──→ 非流式方法 → 返回 Content-Type: application/json
    │
    └──→ 流式方法 → 建立 SSE 连接 → 返回 Content-Type: text/event-stream
```

**SSE 流式传输实现细节**：

**传输机制**：
- 使用 Server-Sent Events (SSE) 协议
- Content-Type: `text/event-stream`
- 每个事件包含一个 JSON-RPC 消息
- 通过 `event` 字段区分事件类型
- 通过 `data` 字段传递 JSON 数据
- 连接保持活跃直到任务完成

**SSE 事件类型**：

| 事件类型 | 说明 | 用途 |
|---------|------|------|
| `message` | JSON-RPC 消息 | 传递请求响应和通知 |
| `done` | 结束标记 | 表示流式传输完成 |
| `error` | 错误消息 | 传递错误信息 |

**进度通知格式**：

```
event: message
data: {"jsonrpc":"2.0","method":"notifications/progress","params":{"progressToken":"token-123","progress":50,"total":100}}
```

**消息通知格式**：

```
event: message
data: {"jsonrpc":"2.0","method":"notifications/message","params":{"level":"info","logger":"tool.web_search","data":"正在搜索..."}}
```

**会话管理**：

MCP 协议支持有状态的会话管理：

```
客户端首次请求
    ↓
服务端创建 Session ID
    ↓
返回 Session ID（在响应头或响应体中）
    ↓
客户端后续请求携带 Session ID
    ↓
服务端根据 Session ID 恢复上下文
```

**SSE 连接保活**：

```
event: ping
data: {"type":"keepalive","timestamp":1234567890}
```

定期发送保活消息（默认 30 秒），防止连接超时。

## 🔐 鉴权层设计方案

### 认证机制

#### 1. API Key 认证

**使用场景**：服务间调用、开发者接入

**认证流程**：
1. 客户端在请求头中携带 API Key
2. 服务端验证 API Key 的有效性
3. 加载 API Key 关联的权限信息
4. 通过验证后继续处理请求

**请求头格式**：
```
Authorization: Bearer sk-xxxxxxxxxxxxxxxx
```

**实现要点**：
- API Key 存储：加密存储在数据库
- Key 生成：使用密码学安全的随机生成器
- Key 轮换：支持定期更换 API Key
- 访问日志：记录所有使用记录

#### 2. JWT Token 认证

**使用场景**：用户身份认证、会话管理

**Token 结构**：
- Header：算法和类型
- Payload：用户 ID、角色、过期时间等
- Signature：签名验证

**认证流程**：
1. 用户登录，服务端签发 JWT
2. 客户端在后续请求中携带 JWT
3. 服务端验证 JWT 签名和有效期
4. 提取用户信息继续处理

**实现要点**：
- 签名算法：RS256 或 HS256
- Token 过期：设置合理的过期时间
- Token 刷新：支持 Refresh Token 机制
- Token 撤销：黑名单机制

#### 3. OAuth 2.0

**使用场景**：第三方应用授权、开放平台

**授权流程**：
- 授权码模式（Authorization Code）
- 隐式授权模式（Implicit）
- 客户端凭证模式（Client Credentials）

| 漏桶 | 平滑输出 | 流量整形 |

#### 多维度限流

**1. 用户级别限流**：
- 每个 API Key 独立限流
- 不同套餐不同限额

- 防止恶意攻击
- 保护系统稳定性

**3. 端点级别限流**：
- 不同接口不同限额
- 核心接口优先保护
- 资源密集型接口严格限流

**4. 模型级别限流**：
- 昂贵模型限流更严
- 保护成本控制
- 防止滥用

```

    ↓

    ↓

    │
    ↓

    │   ...
    └─→ [DONE]  ──→ SSE Done    ──→ 客户端结束
    │

数据层保存（完整对话记录、使用统计）
```

- 工具存储在向量数据库
- 基于查询相似度检索工具
- 动态加载相关工具

**MCP Client 集成**：

Model Context Protocol (MCP) 是一个开放标准，允许 AI 应用连接到外部工具和数据源。通过集成 MCP Client，系统可以：

**MCP 协议概述**：
- **标准协议**：基于 JSON-RPC 2.0 的通信协议
- **双向通信**：支持客户端和服务端双向调用
- **资源访问**：访问文件、数据库、API 等外部资源
- **工具调用**：调用 MCP 服务提供的工具
- **提示词模板**：使用 MCP 服务提供的提示词

**MCP Client 架构**：

| 组件 | 职责 |
|------|------|
| Connection Manager | 管理与 MCP 服务器的连接 |
| Resource Client | 访问 MCP 服务器提供的资源 |
| Tool Client | 调用 MCP 服务器提供的工具 |
| Prompt Client | 获取 MCP 服务器的提示词模板 |
| Transport Layer | 支持 stdio、HTTP（SSE 流式）等传输方式 |



**MCP Client 工作流程**：

```
1. 初始化连接
   ↓
2. 发现可用工具（list_tools）
   ↓
3. Agent 选择需要的工具
   ↓
4. 调用 MCP 工具（call_tool）
   ↓
5. 接收工具执行结果
   ↓
6. 将结果传递给 LLM
   ↓
7. 生成最终响应
```

**配置管理**：

**连接配置**：
- **stdio 模式**：通过标准输入输出与本地进程通信
- **HTTP 模式**：通过 HTTP 与远程 MCP 服务通信（支持 SSE 流式）
- **WebSocket 模式**：长连接实时通信（可选）

**服务发现**：
- 静态配置：在配置文件中预定义 MCP 服务
- 动态注册：运行时注册新的 MCP 服务
- 健康检查：定期检查 MCP 服务可用性

**工具映射**：

将 MCP 工具适配为 Rig Tool Trait：

| MCP 工具字段 | Rig Tool 字段 | 说明 |
|------------|--------------|------|

| description | definition().description | 工具描述 |
| inputSchema | definition().parameters | 参数 Schema |
| - | call() | 调用 MCP 服务执行 |

**错误处理**：
- 连接失败：重试机制、降级策略
- 超时处理：设置合理的超时时间
- 工具不可用：优雅降级、错误提示
- 参数验证：校验参数格式和类型

**安全性**：
- **认证**：支持 API Key、OAuth 等认证方式
- **授权**：控制可访问的 MCP 服务和工具
- **沙箱隔离**：限制 MCP 工具的访问权限
- **审计日志**：记录所有 MCP 工具调用

**性能优化**：
- **连接池**：复用 MCP 连接
- **工具缓存**：缓存工具定义和元数据

## 📐 设计模式与最佳实践

### 1. Trait-Based 设计模式

**问题**：需要支持多种 LLM 提供商，每个提供商 API 不同

**解决方案**：
- 定义统一的 Trait 接口
- 各提供商实现相同 Trait
- 用户代码不关心具体提供商

**优势**：
- 松耦合
- 易于测试（可 mock）
- 易于扩展新提供商

### 2. Builder Pattern

**问题**：Agent 和请求有大量可选配置项

**解决方案**：
- 使用构建器模式
- 提供流畅的链式 API
- 延迟构建对象

**优势**：
- 可读性强
- 链式调用
- 编译时检查
- 灵活配置

### 3. Type State Pattern

**问题**：某些操作有状态依赖（如必须先设置 API key）

**解决方案**：
- 使用类型状态模式
- 在类型系统中编码状态
- 编译时强制状态转换

**优势**：
- 编译时保证正确性
- 防止无效状态
- 零运行时开销

### 4. Error Handling 模式

**错误定义**：
- 使用 `thiserror` 定义错误类型
- 细分错误类别
- 提供上下文信息

**错误传播**：
- 使用 `?` 操作符
- 自动类型转换
- 保留错误链

**错误处理**：
- 模式匹配
- 分类处理
- 降级策略

### 5. 异步并发模式

**并行请求**：
- 使用 `tokio::join!` 或 `try_join!`
- 同时执行多个独立任务
- 等待所有任务完成

**流式处理**：
- 使用 `Stream` trait
- 异步迭代器
- 背压控制

**并发控制**：
- Semaphore 限制并发数
- 任务池管理
- 资源清理

### 6. 依赖注入模式

**通过泛型实现**：
- 将依赖作为泛型参数
- 构造时注入具体实现
- 保持灵活性

**优势**：
- 易于测试
- 灵活替换实现
- 类型安全

## 📝 版本历史

### v0.7 (2025-10-13)

- **MCP Server 对外接口**：新增标准 MCP 协议对外服务端点 `/mcp`
- **JSON-RPC 2.0**：完全遵循 MCP 1.0 标准协议规范
- **SSE 流式传输**：基于 Server-Sent Events 的统一流式传输方案
- **MCP 方法支持**：initialize, tools/list, tools/call, resources/list, resources/read, resources/subscribe, prompts/list, prompts/get, completion/complete
- **工具双向能力**：作为 MCP Client 调用外部服务 + 作为 MCP Server 对外提供服务
- **代理转发**：支持将外部 MCP 请求代理转发到其他 MCP 服务器
- **协议互操作**：任何 MCP 客户端（Python、TypeScript、Rust SDK）都可以直接接入
- **会话管理**：支持有状态的 Session 管理，保持上下文
- **流式进度通知**：支持 notifications/progress 和 notifications/message
- **与 LLM API 协同**：/v1/chat 和 /mcp 两种接口共享工具生态
- **基于 rmcp**：使用 Rust 官方 MCP 库实现，与 rmcp-actix-web 集成

### v0.6 (2025-10-13)

- **新增 MCP Client**：Tools 模块集成 Model Context Protocol 客户端
- **MCP 协议支持**：基于 JSON-RPC 2.0 的标准协议通信
- **工具生态**：支持 Filesystem、Database、Git、Slack 等 MCP 服务
- **传输方式**：支持 stdio、HTTP、WebSocket 多种传输模式
- **安全机制**：认证授权、沙箱隔离、审计日志
- **使用场景**：文件访问、数据库查询、多 MCP 协作等示例
- **架构优化**：更新架构图，展示 MCP Client 在 Tools 模块中的位置

### v0.5 (2025-10-13)

- **新增数据层**：在鉴权层和业务层之间增加数据持久化层
- **数据管理**：对话历史、用户数据、向量存储、配置管理、缓存
- **数据库选型**：PostgreSQL、MongoDB、Qdrant/Milvus、Redis 等
- **数据安全**：加密存储、访问控制、备份恢复、合规审计
- **交互场景**：补充上下文对话、RAG 检索、响应缓存等场景流程图
- **更新流程图**：完善请求处理流程，展示数据层交互
